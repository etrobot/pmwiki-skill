# 运维能力引入时机指南

## 核心原则

**不要过早优化，但也不要等到着火才救**

- Phase 1：能跑就行，手动处理一切
- Phase 2：开始自动化，建立基础可观测性
- Phase 3：完善监控，预防性运维
- Phase 4：智能化运维，自愈能力

---

## 一、定时任务 (Cron Jobs / Scheduled Tasks)

### Phase 1 — 不需要
**原因**：用户量少，手动操作即可

**例外情况**：
- ❌ 不需要：数据清理、报表生成
- ✅ 必须有：证书过期检查（如果用 HTTPS）

---

### Phase 2 — 开始引入基础定时任务
**触发条件**：
- 用户数 > 100
- 有需要定期执行的重复性工作
- 手动操作开始出错或遗漏

**必须引入的定时任务**：

| 任务类型 | 频率 | 目的 | 示例 |
|---------|------|------|------|
| **数据库备份** | 每天 1 次 | 防止数据丢失 | `0 2 * * * pg_dump > backup.sql` |
| **日志轮转** | 每天 1 次 | 防止磁盘占满 | `0 0 * * * logrotate /etc/logrotate.conf` |
| **过期数据清理** | 每周 1 次 | 释放存储空间 | 删除 30 天前的临时文件 |
| **健康检查** | 每 5 分钟 | 确保服务存活 | `*/5 * * * * curl http://localhost/health` |

**技术选型**：
- **本地客户端**：系统 Cron（Linux/Mac）、Task Scheduler（Windows）
- **toC/toB**：Cron + 脚本，或使用 APScheduler（Python）、node-cron（Node.js）

---

### Phase 3 — 完善业务定时任务
**触发条件**：
- 用户数 > 1000
- 有复杂的业务逻辑需要定期执行
- 需要生成定期报表

**新增定时任务**：

| 任务类型 | 频率 | 目的 | 示例 |
|---------|------|------|------|
| **报表生成** | 每天/每周 | 给用户发送周报、月报 | 生成项目进度报告 |
| **数据同步** | 每小时 | 与第三方系统同步数据 | 同步 CRM 数据 |
| **过期提醒** | 每天 1 次 | 提醒用户续费、任务到期 | 发送到期通知邮件 |
| **数据聚合** | 每小时 | 预计算统计数据 | 计算用户活跃度 |
| **缓存预热** | 每天 1 次 | 提升访问速度 | 预加载热门数据到 Redis |

**技术选型**：
- **分布式定时任务**：Celery Beat（Python）、Bull（Node.js）
- **可视化管理**：Airflow、Prefect

---

### Phase 4 — 智能化调度
**触发条件**：
- 定时任务 > 50 个
- 任务之间有复杂依赖关系
- 需要动态调整执行时间

**新增能力**：
- **任务编排**：DAG（有向无环图）管理任务依赖
- **动态调度**：根据系统负载自动调整执行时间
- **失败重试**：自动重试失败任务，指数退避
- **任务监控**：实时查看任务执行状态、耗时

**技术选型**：
- **工作流引擎**：Airflow、Temporal、Prefect
- **分布式调度**：XXL-Job、Quartz

---

## 二、日志系统 (Logging)

### Phase 1 — 最小化日志
**原因**：用户少，直接看控制台输出即可

**必须有的日志**：
- ✅ 错误日志（Error）：记录崩溃、异常
- ✅ 关键操作日志（Info）：用户登录、支付、数据修改
- ❌ 不需要：调试日志（Debug）、性能日志

**技术选型**：
- **本地客户端**：写入本地文件（`app.log`）
- **toC/toB**：标准输出（stdout）+ 云平台自动收集

**日志格式**：
```
[2026-05-08 10:30:45] ERROR - User login failed: Invalid password (user_id=123)
```

---

### Phase 2 — 结构化日志 + 集中收集
**触发条件**：
- 用户数 > 100
- 开始出现难以复现的 Bug
- 需要分析用户行为

**必须引入的能力**：

| 能力 | 目的 | 示例 |
|------|------|------|
| **结构化日志** | 方便查询、分析 | JSON 格式：`{"level":"error","msg":"login failed","user_id":123}` |
| **日志级别** | 区分重要性 | DEBUG < INFO < WARN < ERROR < FATAL |
| **日志收集** | 集中管理多台服务器日志 | Fluentd、Logstash、Vector |
| **日志存储** | 持久化、可搜索 | Elasticsearch、Loki、CloudWatch Logs |
| **日志查询** | 快速定位问题 | Kibana、Grafana、CloudWatch Insights |

**技术选型**：
- **本地客户端**：Winston（Node.js）、Loguru（Python）+ 本地文件
- **toC/toB**：ELK Stack（Elasticsearch + Logstash + Kibana）或 Grafana Loki

**日志保留策略**：
- ERROR 日志：保留 90 天
- INFO 日志：保留 30 天
- DEBUG 日志：保留 7 天

---

### Phase 3 — 日志分析 + 告警联动
**触发条件**：
- 日志量 > 1GB/天
- 需要主动发现问题，而不是等用户反馈

**新增能力**：

| 能力 | 目的 | 示例 |
|------|------|------|
| **日志聚合** | 发现共性问题 | 统计最近 1 小时内出现次数最多的错误 |
| **日志告警** | 异常时自动通知 | ERROR 日志 > 100 条/分钟 → 发送告警 |
| **日志关联** | 追踪请求链路 | Trace ID 关联一次请求的所有日志 |
| **敏感信息脱敏** | 保护用户隐私 | 自动隐藏密码、手机号、身份证号 |

**技术选型**：
- **链路追踪**：OpenTelemetry、Jaeger、Zipkin
- **日志告警**：ElastAlert、Grafana Alerts

---

### Phase 4 — 智能日志分析
**触发条件**：
- 日志量 > 100GB/天
- 人工分析日志成本过高

**新增能力**：
- **异常检测**：AI 自动识别异常日志模式
- **根因分析**：自动关联日志、指标、事件，定位问题根源
- **日志压缩**：智能采样，减少存储成本

**技术选型**：
- **AI 日志分析**：Datadog、New Relic、Splunk

---

## 三、告警系统 (Alerting)

### Phase 1 — 不需要
**原因**：用户少，手动检查即可

**例外情况**：
- ✅ 必须有：服务完全宕机时的告警（Uptime 监控）

**技术选型**：
- **免费工具**：UptimeRobot、Pingdom（免费版）

---

### Phase 2 — 基础告警
**触发条件**：
- 用户数 > 100
- 开始有用户在非工作时间使用
- 出现过因未及时发现问题导致的故障

**必须引入的告警**：

| 告警类型 | 触发条件 | 通知方式 | 示例 |
|---------|---------|---------|------|
| **服务宕机** | HTTP 健康检查失败 | 短信 + 电话 | 服务无响应超过 2 分钟 |
| **错误率飙升** | 错误日志 > 100 条/分钟 | 邮件 + Slack | 登录失败率 > 10% |
| **磁盘空间不足** | 磁盘使用率 > 85% | 邮件 | 数据库磁盘剩余 < 10GB |
| **内存泄漏** | 内存使用率持续上升 | 邮件 | 内存使用率 > 90% 持续 10 分钟 |
| **证书过期** | SSL 证书 < 7 天过期 | 邮件 | HTTPS 证书将于 7 天后过期 |

**技术选型**：
- **本地客户端**：Sentry（崩溃上报）
- **toC/toB**：Prometheus + Alertmanager、CloudWatch Alarms

**告警规则**：
```yaml
# Prometheus 告警规则示例
- alert: HighErrorRate
  expr: rate(http_requests_total{status=~"5.."}[5m]) > 0.1
  for: 5m
  labels:
    severity: critical
  annotations:
    summary: "High error rate detected"
```

---

### Phase 3 — 完善告警体系
**触发条件**：
- 用户数 > 1000
- 有 SLA 承诺（如 99.9% 可用性）
- 告警噪音过多，需要优化

**新增能力**：

| 能力 | 目的 | 示例 |
|------|------|------|
| **告警分级** | 区分紧急程度 | P0（立即处理）、P1（1小时内）、P2（当天处理） |
| **告警聚合** | 减少噪音 | 5 分钟内相同告警只发送 1 次 |
| **告警升级** | 确保问题被处理 | P0 告警 10 分钟未确认 → 通知上级 |
| **告警静默** | 维护期间不发送告警 | 部署期间静默"服务宕机"告警 |
| **告警关联** | 快速定位问题 | 告警附带相关日志、指标链接 |

**告警通知渠道**：
- **P0（紧急）**：短信 + 电话 + Slack
- **P1（重要）**：Slack + 邮件
- **P2（一般）**：邮件

**技术选型**：
- **告警管理**：PagerDuty、Opsgenie、OnCall（Grafana）

---

### Phase 4 — 智能告警 + 自愈
**触发条件**：
- 告警数量 > 100 条/天
- 需要减少人工干预

**新增能力**：
- **智能降噪**：AI 识别误报，自动过滤
- **异常预测**：提前预警潜在问题（如磁盘将在 3 天后满）
- **自动修复**：触发告警后自动执行修复脚本（如重启服务、扩容）
- **告警根因分析**：自动关联多个告警，找出根本原因

**技术选型**：
- **AIOps 平台**：Moogsoft、BigPanda
- **自愈系统**：Kubernetes（自动重启 Pod）、AWS Auto Scaling

## 四、综合决策矩阵

### 按产品类型

| 运维能力 | 本地客户端 | toC 应用 | toB SaaS |
|---------|-----------|---------|---------|
| **定时任务** | Phase 2（本地备份） | Phase 2（数据清理） | Phase 2（数据备份） |
| **日志系统** | Phase 1（本地文件） | Phase 2（集中收集） | Phase 2（集中收集） |
| **告警系统** | Phase 2（崩溃上报） | Phase 2（服务监控） | Phase 2（服务监控） |
| **链路追踪** | 不需要 | Phase 3 | Phase 3 |
| **自动扩容** | 不需要 | Phase 3 | Phase 3 |
| **灾备** | Phase 3（云同步） | Phase 3（多区域） | Phase 3（多区域） |

---

### 按用户规模

| 用户规模 | 定时任务 | 日志 | 告警 | 监控 |
|---------|---------|------|------|------|
| **< 100** | 手动执行 | 本地文件 | Uptime 监控 | 无 |
| **100-1000** | Cron + 脚本 | 结构化日志 + 集中收集 | 基础告警（宕机、错误率） | 基础指标（CPU、内存） |
| **1000-10000** | 分布式调度 | 日志分析 + 告警联动 | 告警分级 + 聚合 | APM（应用性能监控） |
| **> 10000** | 工作流引擎 | AI 日志分析 | 智能告警 + 自愈 | 全链路监控 + AIOps |

---

### 按故障影响

| 故障类型 | 影响范围 | 引入时机 | 优先级 |
|---------|---------|---------|--------|
| **服务宕机** | 所有用户无法使用 | Phase 1 | P0 |
| **数据丢失** | 用户数据永久丢失 | Phase 2 | P0 |
| **性能下降** | 用户体验变差 | Phase 2 | P1 |
| **功能异常** | 部分功能不可用 | Phase 2 | P1 |
| **UI 错误** | 界面显示异常 | Phase 3 | P2 |

## 五、实施建议

### 1. 优先级排序（Phase 2 必须做）

```
1. 数据库备份（防止数据丢失）
2. 服务宕机告警（确保及时响应）
3. 错误日志收集（快速定位问题）
4. 磁盘空间监控（防止服务崩溃）
5. 日志轮转（防止磁盘占满）
```

### 2. 成本控制

| 阶段 | 月成本预算 | 建议方案 |
|------|-----------|---------|
| Phase 1 | $0 | 免费工具（UptimeRobot、Sentry 免费版） |
| Phase 2 | $50-200 | 云平台自带监控（CloudWatch、Azure Monitor） |
| Phase 3 | $500-2000 | 自建 ELK + Prometheus 或使用 Datadog Starter |
| Phase 4 | $5000+ | 企业级 APM（Datadog、New Relic）+ AIOps |

### 3. 团队能力要求

| 阶段 | 需要的角色 | 技能要求 |
|------|-----------|---------|
| Phase 1-2 | 开发工程师 | 会写脚本、配置 Cron |
| Phase 3 | 开发 + 运维 | 熟悉 Prometheus、ELK、Docker |
| Phase 4 | SRE 团队 | 熟悉 Kubernetes、分布式系统、AIOps |

## 六、常见错误

### ❌ 过早优化
- **错误**：Phase 1 就搭建完整的 ELK + Prometheus + Grafana
- **后果**：浪费时间，增加维护成本
- **正确做法**：先用云平台自带的监控，够用就行

### ❌ 过晚引入
- **错误**：Phase 3 了还没有数据库备份
- **后果**：数据丢失，无法恢复
- **正确做法**：Phase 2 必须有备份 + 告警

### ❌ 告警过多
- **错误**：所有 ERROR 日志都发送告警
- **后果**：告警疲劳，真正的问题被忽略
- **正确做法**：只对影响用户的问题发送告警

### ❌ 日志不结构化
- **错误**：日志格式混乱，无法查询
- **后果**：排查问题困难，浪费时间
- **正确做法**：Phase 2 开始使用 JSON 格式日志

## 七、检查清单

### Phase 1 → Phase 2 必须完成

- [ ] 数据库每天自动备份
- [ ] 服务宕机时发送告警（邮件/短信）
- [ ] 错误日志写入文件，保留 30 天
- [ ] 磁盘使用率 > 85% 时发送告警
- [ ] SSL 证书过期前 7 天发送提醒

### Phase 2 → Phase 3 必须完成

- [ ] 日志集中收集到 Elasticsearch / Loki
- [ ] 告警分级（P0/P1/P2）
- [ ] 关键业务指标监控（QPS、延迟、错误率）
- [ ] 定时任务失败时发送告警
- [ ] 链路追踪（Trace ID 贯穿所有日志）

### Phase 3 → Phase 4 必须完成

- [ ] 告警聚合 + 降噪
- [ ] 异常检测（AI 识别异常模式）
- [ ] 自动扩容（根据负载自动调整资源）
- [ ] 灾备演练（每季度 1 次）
- [ ] SLA 监控（可用性、性能指标）

## 八、总结

| 运维能力 | Phase 1 | Phase 2 | Phase 3 | Phase 4 |
|---------|---------|---------|---------|---------|
| **定时任务** | ❌ 不需要 | ✅ 基础任务（备份、清理） | ✅ 业务任务（报表、同步） | ✅ 智能调度 |
| **日志** | ✅ 本地文件 | ✅ 结构化 + 集中收集 | ✅ 分析 + 告警联动 | ✅ AI 分析 |
| **告警** | ✅ Uptime 监控 | ✅ 基础告警（宕机、错误） | ✅ 分级 + 聚合 | ✅ 智能降噪 + 自愈 |
| **监控** | ❌ 不需要 | ✅ 基础指标（CPU、内存） | ✅ APM + 链路追踪 | ✅ 全链路 + AIOps |

**核心原则**：
1. **Phase 1**：能跑就行，手动处理
2. **Phase 2**：自动化基础运维，建立可观测性（**最关键的阶段**）
3. **Phase 3**：完善监控，预防性运维
4. **Phase 4**：智能化运维，自愈能力
